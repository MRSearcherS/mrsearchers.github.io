## Проект

Проект — статический сайт про жёлтую Toyota MR-S 2000 года.

Машина условно называется «Желтик».

Сайт рассказывает о машине, её истории, особенностях, фотографиях и может использоваться как презентационная страница автомобиля.

Основной публичный домен:

https://mrs.spb.ru/

Именно этот адрес считать финальным публичным адресом сайта.

## Пользователь и язык общения

Пользователь предпочитает русский язык.

Ассистент должен отвечать по-русски, если пользователь пишет по-русски.

PowerShell

## Архитектура сайта

Сайт статический.

Была выбрана архитектура с исходниками в папке src.

Основные файлы:

- src/template.html
- src/build.js
- src/locales/ru.json
- src/locales/en.json
- src/locales/zh.json

Сборка выполняется через Node.js.

Команда сборки:

    node src/build.js

После сборки обновляются HTML-файлы сайта:

- index.html
- en/index.html
- zh/index.html

## Важные технические проблемы, которые уже встречались

### 1. Потеря контекста ассистентом

Проблема: длинная переписка частично выпадает из контекста, ассистент начинает забывать детали или противоречить сам себе.

Решение: вести набор файлов:

- PROJECT_CONTEXT.md
- CURRENT_CODE.md
- WORK_LOG.md
- OPEN_TASKS.md
- DECISIONS.md

---

### 2. Git и GitHub

В проекте уже были сложности с Git/GitHub, включая push, rebase, конфликты и публикацию через GitHub Pages.

Правило: перед опасными действиями сначала просить пользователя показать:

    git status

Не предлагать команды, которые меняют историю или удаляют изменения, без объяснения риска.

---

### 3. Изображения

Были проблемы с именами файлов изображений, включая ситуацию с двойным расширением вроде:

.jpg.jpg

Правило: не предполагать имена файлов. Сначала проверять реальные имена и пути.

---

### 4. JSON и кодировка

Были проблемы с JSON-файлами и/или кодировкой.

Возможные причины:

- BOM;
- невалидный JSON;
- лишние запятые;
- неправильные кавычки;
- случайные символы в начале файла.

Правило: при ошибках сборки проверять валидность JSON и кодировку UTF-8.

---

### 5. Опасные regex-команды

Был негативный опыт: ассистент предложил слишком широкий regex/команду удаления HTML-блока, что могло удалить лишний код.

Правило:

- не использовать агрессивные массовые удаления по regex без предварительного просмотра;
- сначала показать найденный фрагмент;
- уточнить границы блока;
- предложить безопасный ручной вариант;
- только потом давать команду.

---

### 6. Блок qr-sign

Обсуждался блок qr-sign.

Для проверки остатков блока/переменных использовалась команда:

    Select-String -Path "src\template.html","index.html","en\index.html","zh\index.html" -Pattern "qr-sign|qrTitle|qrText|undefined"

Если блок удаляется из шаблона, после этого нужно запускать:

    node src/build.js

## Предпочтения пользователя

- Отвечать по делу.
- Не додумывать отсутствующий контекст.
- Если данных не хватает — сначала спросить.
- Давать команды под Windows PowerShell.
- Не давать устаревший код, если он больше не используется.
- Если предлагается изменение кода — указывать файл и место изменения.
- Если есть риск повредить проект — сначала объяснить риск.
- Не делать вид, что ассистент помнит то, чего нет в текущем контексте.

## Как ассистент должен работать с проектом

Перед рекомендациями по коду желательно запросить актуальные файлы:

- src/template.html
- src/build.js
- нужный JSON из src/locales
- текущий index.html, если проблема именно в собранной версии

Если пользователь прислал CURRENT_CODE.md, использовать его как источник актуального кода.

Если есть расхождение между памятью и текущим кодом, текущий код важнее.
